# Deepseek MaxText to vLLM Converter Plan

## Objective
Rewrite and optimize the `DeepSeekV3ToVLLMConverter` in `bench_weight_sync.py` so that it directly generates **all** keys expected by vLLM in the `vllm_state` dictionary. This eliminates the need for dynamic quantization during the `llm_state` assignment loop, ensures all scales and inverse scales are correctly populated, and prevents vLLM from executing with uninitialized (dummy) weights.

## 1. Eliminate Dynamic Assignment Loop Quantization
**Current State:** The assignment loop at the bottom of `bench_weight_sync.py` intercepts FP8 keys, calls `tpu_quantize_tensor` on the fly, and assigns `weight_scale`. However, it fails to populate `weight_scale_inv` and masks exactly which keys the converter actually produced.
**Action:** 
- Remove the dynamic quantization logic from the assignment loop.
- The assignment loop should *only* perform `_get_reshard_fn` transfers. 
- `vllm_state` must be the absolute source of truth. If a key is in `llm_state`, it MUST be pre-calculated and present in `vllm_state`.

## 2. Explicit FP8 Quantization in the Converter
**Action:** Modify the `DeepSeekV3ToVLLMConverter` to explicitly quantize all dense linear layers to FP8 and generate both `weight_scale` and `weight_scale_inv`.

For every dense FP8 linear layer (`q_b_proj`, `o_proj`, `fused_qkv_a_proj`, `down_proj`, `gate_up_proj`):
1. Cast the FP32/BF16 weights to FP32.
2. Call `tpu_quantize_tensor(jnp.float8_e4m3fn, weight, axis=-1)`.
3. Compute `scale_inv = 1.0 / scale`.
4. Store **three** keys in `vllm_state`:
   - `[prefix].weight` -> The FP8 quantized array.
   - `[prefix].weight_scale` -> The `scale` array.
   - `[prefix].weight_scale_inv` -> The `scale_inv` array.

## 3. Resolve MLA Attention Discrepancies
**Current State:** vLLM's `llm_state` expects `kv_b_proj.weight`, `mla_attn._k_scale`, `_prob_scale`, `_q_scale`, `_v_scale`, and the newly injected `W_UK_T` / `W_UV`.
**Action Items:**
1. **`kv_b_proj` cleanup:** Even though `mla_attention.py` deletes `kv_b_proj.weight`, it still appears in `llm_state`. The converter should either supply a dummy tensor for it (since it's bypassed in the forward pass) or `bench_weight_sync.py` should explicitly pop it from `llm_state` before the sync loop.
2. **`W_UK_T` & `W_UV` scales:** Ensure that `W_UK_T_scale` and `W_UV_scale` (and their `_inv` counterparts if requested by vLLM) are produced by the converter with the correct shapes `(1, num_heads, 1, rank)` and `(1, num_heads, 1, v_head)`.
3. **Activation Scales:** Create scalar tensors (`jnp.ones(1, dtype=jnp.float32)`) for the internal MLA scaling factors:
   - `mla_attn.mla_attn._q_scale`
   - `mla_attn.mla_attn._k_scale`
   - `mla_attn.mla_attn._v_scale`
   - `mla_attn.mla_attn._prob_scale`

## 4. Handle MoE Layers Accurately
**Action Items:**
1. **Routed Experts (`w13_weight`, `w2_weight`):** 
   - The current converter already processes these using `_process_routed`. 
   - Ensure the generated keys match exactly: `w13_weight`, `w13_weight_scale_inv`, `w2_weight`, and `w2_weight_scale_inv`. (Verify if `weight_scale` non-inv is also expected by vLLM for MoE).
2. **Shared Experts (`gate_up_proj`, `down_proj`):**
   - Apply the explicit dense FP8 quantization (from Step 2) to the shared experts to generate their `weight`, `weight_scale`, and `weight_scale_inv`.

## 5. Ignore Runtime-Generated Caches
**Action Item:** The `rotary_emb.cos_sin_cache` is a precomputed RoPE cache generated by vLLM during initialization. It does not come from the MaxText checkpoint. 
- Modify the `[KEY_CHECK]` diagnostic and the assignment loop to explicitly ignore `rotary_emb.cos_sin_cache`.

## 6. Discovering the Source of Truth Keys
**Action:** To ensure we generate the exact keys vLLM expects, we will use the `llm_state` dictionary extracted directly from the initialized vLLM engine as the absolute source of truth.
- **Mechanism:** vLLM initializes using the `pathways_dummy` loader, which populates the model architecture with dummy tensors. The keys of `llm_state` (`llm.llm_engine.model_executor.driver_worker.model_runner.state`) at this exact moment represent the precise parameter names and expected shapes for the entire distributed model.
- **Validation (Final Goal):** We must guarantee that **no keys** from the original `llm_state` (the vLLM state) are left out or skipped by the MaxText converter (excluding explicitly ignored caches like `rotary_emb.cos_sin_cache`). We will log the `[KEY_CHECK]` diagnostic, which computes `missing_in_vllm = set(llm_state.keys()) - set(vllm_state.keys())`. Our goal is to drive the `missing_in_vllm` list to exactly zero.

## 7. Validation Strategy
1. **Apply the modifications** to `bench_weight_sync.py`.
2. **Build and push** the new Docker image (`20260413m`).
3. **Deploy** the `deepseek_nextdev_bench.yaml` workload.
4. **Monitor** the logs. The `[KEY_CHECK] llm_state keys NOT in vllm_state` output should shrink to zero (or only list known ignored keys like `cos_sin_cache` and `kv_b_proj`).
5. **Verify** that generation completes successfully without returning NaN or garbage text.
## 8. Current Known Error
**Action Item:** The deployment was crashing during the forward pass with an `AssertionError` in `gmm_v2.py` checking `rhs_scale.shape == (size_group, num_quant_blocks, 1, size_n)`.
**Diagnosis:** The issue was caused by two intertwined shape mismatches during the explicit FP8 quantization:
1. `down_proj` (`w2_weight`) was incorrectly transposed to `(E, d_inner, d_model)` prior to quantization. Since vLLM's `pathways_dummy` loader relies on the shape of `w2_weight` to extract `hidden_size` and `intermediate_size` (`d_model` and `d_inner` respectively), the swapped dimensions caused `process_moe_weights` to misinterpret the network topology, leading to mismatched padding configurations. We fixed this by correctly transposing `down` to `(E, d_model, d_inner)` before quantization.
2. The quantization axes were incorrect. For per-channel quantization, the reduction axis must be the contraction dimension. For `gate_up` (`w13`), it is `d_model` (`axis=2`). For `down` (`w2`), it is `d_inner` (`axis=2` after the correct transpose). Both must then have `expand_dims(axis=(2,3))` applied to correctly broadcast as `(E, 1, 1, size_n)`.

**Status:** FIXED. The deployment now cleanly initializes, the `[KEY_CHECK]` shows exactly zero missing keys (ignoring caches), and the XLA compilation for the generation test successfully triggers without JIT assertion errors.
