## Round 1 (2026-06-11): tmem deep-dive article

The hardest part was correctly tracing which Pipeline class the ds-proxy config actually uses. The config has `num_pipeline_repeats=2` (circular) but `pipeline_fsdp_ag_per_repeat=false`, so it uses the `Pipeline` class, not `CircularPipeline`. This means the local-ops optimization applies. I also had to be careful about the `_shift_right` vs `_rotate_right` distinction and why `lax.slice` is chosen over `slice_in_dim` for gradient properties. One concern: the per-optimization savings estimates in the reference doc don't add up exactly to 9.5 GB because buffer assignment is a global optimization, and I called this out explicitly rather than trying to force the numbers. Next time: compute the actual activation sizes more precisely from the config dimensions rather than relying on the reference doc's rough estimates.

## Round 2 (2026-06-11): tmem deep-dive revision

Five fixes from verifiers: (1) VMA = "Varying Manual Axes" not "Verified Memory Annotation" -- classic confabulation trap, always web-search acronym expansions; (2) DeepSeek V2 has 60 layers, V3 has 61, MLA = "Multi-head Latent Attention"; (3) SPMD partitioner runs during `.compile()` not `.lower()` -- subtle but critical; (4) `meta.remove_axis` is NOT removed in the current codebase, it's still at lines 622 and 684 of pipeline.py, so Section 7.3 had to be entirely removed; (5) microbatch size: `micro_batch_size_to_train_on = num_devices * per_device_batch_size = 8 * 24 = 192`, so `pipeline_microbatch_size = 192/64 = 3`. The savings table was also completely restructured: instead of fake GB estimates that summed to 2x the total, I used qualitative ranking (largest/second/moderate/small) and framed them as non-additive standalone impacts. This avoids the impossible arithmetic while being honest about what we know vs. what we'd need profiling isolation to determine.
